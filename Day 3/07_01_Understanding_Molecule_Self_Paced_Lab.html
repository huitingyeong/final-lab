1. Start Lab Environment

You need to run following steps if your lab environment has shut down. To conserve resources, the lab environment shuts down automatically after eight hours.

1.1. Start Environment After Shut Down

  1. Go to the OPENTLC lab portal and use your OPENTLC credentials to log in.

  2. Navigate to Services → My Services (this should be the screen shown right after logging in).

  3. In the list of your services, select your lab environment.

  4. Select App Control → Start to start your lab environment.

  5. Select Yes at the Are you sure? prompt.

  6. On the bottom right, click Submit.

    After a few minutes, expect to receive an email letting you know that the lab environment has been started.

1.2. Connect bastion Node

  1. Bastion node is the main host which is accssible via ssh from public network. You will be using this host for running commands and accessing other host available in your environment. You will use ssh to connect bastion node, using the login details provided in third email.

    [laptop]$ ssh <OPENTLC-USER-ID>@bastion.${GUID}.${SUBDOMAIN}

  2. Switch to devops user. The User devops is service account in bastion node who has complete sudoer privilege to run all commands with no sudo password.

    [bastion]$ sudo su - devops

1.3. Refresh Satellite Manifest

  1. After restart of your classroom environment need to run following Lab Manager command to refresh satellite manifest.

    $ lab refresh -i satellite

3. Molecule

Ansible Molecule project is designed to aid in the development and testing of Ansible roles. Molecule provides support for testing with multiple instances, operating systems and distributions, virtualization providers, test frameworks and testing scenarios. Molecule encourages an approach that results in consistently developed roles that are well-written, easily understood and maintained.

3.1. Prepare Node

In this section, you will install OS dependencies and create python virtual environment for molecule.

  1. Before you start, you need to connect with node designated for the molecule lab.

    $ ssh node.example.com

  2. You need to install following packages to prepare node for molecule.

    • ansible

    • podman

    • tree

    • gcc

    • python3-pip

    • python3-devel

    • python3-libselinux

    • openssl-devel

    [node]$ sudo dnf install -y ansible podman tree gcc python3-pip python3-devel python3-libselinux openssl-devel

  3. It is highly recommended that you install molecule in a python virtual environment. So you follow the step to setup virtual environment.

    [node]$ mkdir $HOME/venvs
[node]$ python3.8 -m venv $HOME/venvs/molecule

  4. To activate python virtual environment, run following command.

    [node]$ source $HOME/venvs/molecule/bin/activate
(molecule)[node]$
    Python virtual env activation process changes the terminal prompt name. which is indication that you are in python virtual environment. To deacative, simply execute deactivate.

3.2. Install Molecule

  1. You can now install molecule in python virtual enviroment using pip cli but there is issue with latest version of cryptography python library so we need to install specific version of cryptography-3.2.1 and the molecule-3.5.2 as follows.

    (molecule)[node]$ pip install molecule==3.5.2 cryptography==3.2.1

  2. You can run following command with flag to display molecule version.

    (molecule)[node]$ molecule --version
    Version Output
    molecule 3.5.2 using python 3.8
    ansible:2.12.1
    delegated:3.5.2 from molecule

  3. Molecule is simple command line tool like other linux commands so you can use --help to explore its options.

    (molecule)[node]$ molecule --help

3.3. Install Molecule Driver

Molecule uses drivers to bring up Ansible ready hosts to operate on. It uses the “delegated” aka “DIY” driver by default. Other drivers can be installed separately from PyPI repository, such as the molecule-podman, molecule-vagrant, molecule-azure or molecule-hetzner.

  1. You will be using molecule-podman-1.1.0 driver to bring ansible host up and the yamllint-1.26.3 to check syntaxes. So you need to install the driver and yamllint in python virtual environment as follows.

    (molecule)[node]$ pip install molecule-podman==1.1.0 yamllint==1.26.3

  2. You can list all installed molecule drivers after the installation as follows.

    (molecule)[node]$ molecule drivers
    Drivers Output

4. Inspect Default Scenario

In this section, you are goint to inspect all of files which got created during initialization of the role in side the molecule default scenario.

4.1. molecule.yml

  1. The molecule.yml is central configuration entrypoint for Molecule which can be configured according to your use case. It is a YAML file whose keys represent the high level components that Molecule provides.

    (molecule)[node]$ cat common/molecule/default/molecule.yml
---
dependency:     (1)
  name: galaxy
driver:         (2)
  name: podman
platforms:      (3)
  - name: instance
    image: docker.io/pycontribs/centos:8
    pre_build_image: true
provisioner:    (4)
  name: ansible
verifier:       (5)
  name: ansible
    Callouts
    1The Dependency manager. By default molecule uses galaxy to resolve role dependencies if any.
    2The Driver provider. By default molecule uses docker to delegate the task of creating instances. This can be set to another supported driver like podman.
    3The Platforms definitions. Molecule uses platform values to create instances/container, set their name, and configure to which group each instance belongs. You can also provide specific image and many more settings.
    4The Provisioner. Molecule only provides an ansible provisioner which manages the life cycle of the instance based on the configuration.
    5The Verifier framework. By default molecule uses ansible to provide a way to write specific state checking tests (such as deployment smoke tests) on the target instance.

4.2. converge.yml

  1. The converge.yml file is the playbook that contains the call for your role. Molecule will invoke this playbook with ansible-playbook and run it against an instance created by the driver.

    (molecule)[node]$ cat common/molecule/default/converge.yml
---
- name: Converge
  hosts: all            (1)
  tasks:                (2)
    # replace these tasks with whatever you find suitable to test
    - name: Copy something to test use of synchronize module
      ansible.posix.synchronize:
        src: /etc/hosts
        dest: /tmp/hosts-from-controller
      # synchronize module works with podman connections only with 2.11+ so
      # we skip running it on unsupported platforms.
      when: "ansible_version.full is version_compare('2.11', '>=')"
    - name: "Include common"
      include_role:     (3)
        name: "common"
    Callouts
    1List of ansible hosts or groups against which molecule will run converge.yml playbook.
    2Tasks of the playbook, You can prepare converge.yml playbook according to your requirements.
    3Call for your role.

4.3. verify.yml

  1. verify.yml is the ansible playbook used for testing with the default verifier which is ansible. This allows you to write specific tests against the state of the container after your role has finished executing.

    (molecule)[node]$ cat common/molecule/default/verify.yml
---
# This is an example playbook to execute Ansible tests.

- name: Verify
  hosts: all          (1)
  gather_facts: false
  tasks:              (2)
  - name: Example assertion
    assert:
      that: true
    Callouts
    1List of ansible hosts or groups against which molecule will run converge.yml playbook.
    2Tasks of the playbook, You can add tasks to check stat of the instances which gets changed after your role execution.

5. Run Molecule

In this section, you will run molecule with various test Sequences

  1. Due to bug current version of molecule-podman doesn’t read collection from molecule custom collections path. So as workaround you need to install following ansible collections in default location.

    (molecule)[node]$ ansible-galaxy collection install containers.podman
(molecule)[node]$ ansible-galaxy collection install ansible.posix

5.1. Create Sequence

  1. In create sequence by default molecule installs role dependencies, create instances/cotainers, and prepare them (run pre configuration task) if prepare playbook is configured in the default scenario. You can run molecule create sequence as follows.

    (molecule)[node]$ cd $HOME/test-project/roles/common
(molecule)[node]$ molecule create
    Create sequence output

  2. You can list all of the created instances in create sequence as follows.

    (molecule)[node]$ molecule list
    List instance output

  3. Molecule provides login flag which you can use to login directly to any of the instances created in create sequence as follows.

    (molecule)[node]$ molecule login -h instance
INFO     Running default > login

[root@instance /]#

  4. You can logout from the instance using exit command like you logout from other linux host.

    [root@instance /]# exit

5.2. Converge Sequence

In converge sequence by default molecule installs role dependencies, create, prepare, and execute converge.yml playbook which includes role on the created instances. To run molecule converge sequence, you need to do following steps.

  1. The role common which you have initialized in previous step, has nothing in it. So you need to add following task in the role before you run converge sequence. As follows.

    (molecule)[node]$ cat > tasks/main.yml <<EOF
---
- name: Configure motd
  copy:
    content: "Welcome to {{ inventory_hostname }}"
    dest: /etc/motd
EOF

  2. Now, you can run molecule converge sequence to test the role as follows.

    (molecule)[node]$ molecule converge
    Converge Output

5.3. Verify Sequence

In verify sequence molecule executes verify.yml playbook which tests against the state of the container after your role has finished executing. To run molecule verify sequence, you need to do following steps.

  1. The role common does not have useful tasks in molecule/default/verify.yml playbook to test state. So you need to add following task in the role before you run converge sequence. As follows.

    (molecule)[node]$ cat > molecule/default/verify.yml <<EOF
---
- name: Verify
  hosts: all
  gather_facts: false
  tasks:
    - name: get content
      slurp:
        src: "/etc/motd"
      register: remote_motd

    - name: Print remote motd
      debug:
        msg: "{{ remote_motd.content | b64decode }}"

    - name: verify motd content
      assert:
        that: '"Welcome to" in remote_motd.content | b64decode'
EOF

  2. Now, you can run molecule verify sequence to verify instance state as follows.

    (molecule)[node]$ molecule verify
    Verify Output

5.4. Destroy Sequence

  1. In destroy sequence molecule deletes all of the instances/cotainers which it created as per default scenario configuration. You can run molecule destroy sequence as follows.

    (molecule)[node]$ molecule destroy
    Destroy output

  2. Now, If you run molecule list command then created field will be set to false in the output which mean none of the instance has been created yet.

    (molecule)[node]$ molecule list
    List output

5.5. Test sequence

  1. Molecule provides commands for manually managing the lifecycle of the instance, scenario, development and testing tools. However, we can also tell Molecule to manage this automatically within a Scenario sequence. We can run full lifecycle sequence as follows.

    (molecule)[node]$ molecule test
    Molecule full test sequence destroys the instance at the end. However we can pass --destroy=never option so molecule will not destroy instance and we can use this instance to test failed task if any. The directory in which the molecule.yml resides is the Scenario’s directory.
