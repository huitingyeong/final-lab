1. Start Lab Environment

You need to run following steps if your lab environment has shut down. To conserve resources, the lab environment shuts down automatically after eight hours.

1.1. Start Environment After Shut Down

  1. Go to the OPENTLC lab portal and use your OPENTLC credentials to log in.

  2. Navigate to Services → My Services (this should be the screen shown right after logging in).

  3. In the list of your services, select your lab environment.

  4. Select App Control → Start to start your lab environment.

  5. Select Yes at the Are you sure? prompt.

  6. On the bottom right, click Submit.

    After a few minutes, expect to receive an email letting you know that the lab environment has been started.

1.2. Connect bastion Node

  1. Bastion node is the main host which is accssible via ssh from public network. You will be using this host for running commands and accessing other host available in your environment. You will use ssh to connect bastion node, using the login details provided in third email.

    [laptop]$ ssh <OPENTLC-USER-ID>@bastion.${GUID}.${SUBDOMAIN}

  2. Switch to devops user. The User devops is service account in bastion node who has complete sudoer privilege to run all commands with no sudo password.

    [bastion]$ sudo su - devops

1.3. Refresh Satellite Manifest

  1. After restart of your classroom environment need to run following Lab Manager command to refresh satellite manifest.

    $ lab refresh -i satellite

2. Overview

This lab is in continuation of the previous lab so you will use some of the settings and resources which you have done already. In this lab you have been provided a mycollections.threetierapp collection which includes many roles inside the collection.

As developer, you need to test one of the role called apache from collection using molecule. This time you will configure molecule to run test against OpenStack instances instead of containers. To test you will add molecule scenario in the existing role as per following requirements.

3. Prepare Node

  1. Before you start, You need copy OpenStack authentication file from bastion which is needed by molecule to provision OpenStack instances. You can find the auth file in /etc/openstack/clouds.yaml in bastion and copy to your node as follows.

    $ ansible node.example.com -m file -a "state=directory dest=/etc/openstack" -b

$ ansible node.example.com -m copy -a "src=/etc/openstack/clouds.yaml dest=/etc/openstack/clouds.yaml" -b

  2. Copy GUID variable in node.example.com. This varialbe will replace GUID in some of the molecule sequence playbooks.

    $ ansible node.example.com -m copy -a "content='export GUID=${GUID}' dest=/home/cloud-user/.bashrc" -b

  3. Now connect to node.example.com.

    $ ssh node.example.com

  4. You need to activate python virtual environment created in previous lab, as follows.

    [node]$ source $HOME/venvs/molecule/bin/activate

  5. You need to install following python library which are required to configure molecule so it can provision OpenStack instance to test role.

    (molecule)[node]$ pip install ansible-core==2.12.2 molecule-openstack==0.3  openstacksdk==0.61.0

  6. You can list all of the installed molecule drivers as follows.

    (molecule)[node]$ molecule drivers
    Drivers output

4. Prepare Role

In this part of the lab, you will add molecule default scenario in apache role and configure molecule in a such way so it can provision OpenStack instances to execute role against them.

4.1. Clone Project

  1. You can clone molecule-project project from gitlab which contains mycollections.threetierapp collection. You can find apache role inside the collection. As per the lab you need to test apache role with molecule.

    (molecule)[node]$ cd $HOME
(molecule)[node]$ git clone http://bastion.example.com:5050/devops/molecule-project.git

4.2. Add Molecule Scenario

  1. You can add molecule scenario (which is a self-contained directory containing everything necessary for testing the role in a particular way) to test the apache role as follows

    (molecule)[node]$ cd $HOME/molecule-project/collections/ansible_collections/mycollections/threetierapp/roles/apache

(molecule)[node]$ molecule init scenario -d openstack

4.3. Update molecule.yml

  1. You need to update molecule.yml in default molecule scenario so that it provisions one app1 (backend_servers) and one appdb (database_servers) OpenStack instances as per minimume requirement of the role to test.

    (molecule)[node]$ cat > molecule/default/molecule.yml <<EOF
---
dependency:
  name: galaxy
driver:
  name: openstack                       (1)
platforms:
  - name: app1                          (2)
    image: rhel-8.3                     (3)
    flavor: GLOBAL-CPU_4_Memory_8192_Disk_30_flavor (4)
    network: ${GUID}-default-network    (5)
    ssh_user: cloud-user                (6)
    groups:
      - backend_servers                 (7)
  - name: appdb                         (2)
    image: rhel-8.3                     (3)
    flavor: GLOBAL-CPU_4_Memory_8192_Disk_30_flavor (4)
    network: ${GUID}-default-network    (5)
    ssh_user: cloud-user                (6)
    groups:
      - database_servers                (7)
provisioner:
  name: ansible
  env:
    ANSIBLE_CONFIG: /home/cloud-user/molecule-project/ansible.cfg (8)
verifier:
  name: ansible
EOF
    Callouts
    1Molecule driver.
    2OpenStack instance name.
    3Instance image.
    4OpenStack flavor.
    5OpenStack network.
    6Ssh user for instances.
    7Ansible inventory groups variable.
    8Ansible Configuration file.

4.4. Update prepare.yml

  1. You need to update prepare.yml so that it can prepare your OpenStack instances upto the level from where your role’s tasks can be executed.

    $ cat > molecule/default/prepare.yml  <<EOF
---
- name: Prepare
  hosts: all
  gather_facts: false
  become: true
  vars:
    satellite_ca_cert_url: http://satellite.example.com/pub/katello-ca-consumer-latest.noarch.rpm
    satellite_activationkey: gpte-labs-rhel8
    satellite_org: prod
  tasks:
    - name: install katello-ca-consumer package
      ansible.builtin.dnf:
        name: "{{ satellite_ca_cert_url }}"
        state: present
        disable_gpg_check: true

    - name: register system and attach subs
      community.general.redhat_subscription:
        state: present
        activationkey: "{{ satellite_activationkey }}"
        org_id: "{{ satellite_org }}"
EOF

4.5. Update converge.yml

  1. As per molecule.yml it creates two instances one for backend_servers and another for database_servers but you need to run your role only on backend_servers since it configures only application so modify molecule/default/converge.yml as follows.

    (molecule)[node]$ cat > molecule/default/converge.yml <<EOF
---
- name: Converge
  hosts: backend_servers (1)
  become: yes
  tasks:
    - name: "Include apache"
      include_role:
        name: "apache"
EOF

4.6. Update verify.yml

  1. The default molecule scenario does not have useful tasks in molecule/default/verify.yml playbook to verify the state of the instances. So you can update verify.yml As follows.

    (molecule)[node]$ cat > molecule/default/verify.yml <<EOF
---
- name: Verify
  hosts: backend_servers
  gather_facts: false
  tasks:
  - name: Access backend server
    uri:
      url: http://localhost:8080
      return_content: true
    register: web_output

  - name: Example assertion
    assert:
      that: '"Enter User Name" in web_output.content'
EOF

5. Testing Role

In this section, you will run molecule to provision OpenStack instances, execute role against the instances and you will also verify instances state as per the molecule default scenario settings and configurations which you did in earlier steps.

  1. Due to bug current version of molecule-podman doesn’t read collection from molecule custom collections path. So as workaround you need to install following ansible collections in default location.

    (molecule)[node]$ cat > $HOME/molecule-project/requirements.yml <<EOF
collections:
  - containers.podman
  - ansible.posix
  - community.general (1)
  - openstack.cloud   (2)
EOF
(molecule)[node]$ ansible-galaxy collection install -r $HOME/molecule-project/requirements.yml

5.1. Create Sequence

  1. You can run molecule create sequence to provision and prepare OpenStack instances as follows.

    (molecule)[node]$ molecule create
    Create sequence output

5.2. Converge Sequence

  1. You can run molecule converge sequence to execute your apache role as follows

    (molecule)[node]$ molecule converge
    Converge Output

5.3. Verify Sequence

  1. You run molecule verify sequence to check the state of the OpenStack instances after your role has finished executing. As follows.

    (molecule)[node]$ molecule verify
    Verify Output

5.4. Destroy Sequence

  1. You can run molecule destroy sequence to deletes all of the provisioned OpenStack instances as follows.

    (molecule)[node]$ molecule destroy

5.5. Test sequence

  1. After running each molecule sequence one by one. Now you can run full lifecycle role test sequence as follows.

    (molecule)[node]$ molecule test
    Molecule full test sequence destroys the instance at the end. However we can pass --destroy=never option so molecule will not destroy instance and we can use this instance to test failed task if any. The directory in which the molecule.yml resides is the Scenario’s directory.
